# Twitter Chrome MCP Fetch Design

## Goal

Replace the current Apify-based X/Twitter fetch path with a Chrome MCP-based daily fetch flow that reads from the user's logged-in Chrome session, saves tweet text and images into project-owned HTML archives under `data/`, and continues producing `data/feed-x.json` for the existing digest and video pipeline.

## Scope

This design only changes the X/Twitter ingestion path.

- Podcasts and blogs remain RSS-based.
- The downstream digest, script, audio, and video pipeline should keep consuming `data/feed-x.json`.
- The new X fetch must not save the raw page HTML from x.com.
- The new X fetch must save tweet data and tweet images, then render project-owned HTML files from that structured data.

## Constraints

- The fetch runs on the local machine and can rely on the user's already logged-in Chrome session.
- X/Twitter data must be read through the Chrome MCP path, not through Apify.
- The fetch is daily and should only keep tweets in the current daily window.
- Replies are excluded.
- The code should tolerate per-account failures and continue fetching other accounts.
- The codebase currently has no automated tests, so verification must rely on targeted command runs and artifact inspection.

## Current State

Today, [fetch-feeds.js](/Users/alexmini/Codex/oneAi/scripts/fetch-feeds.js) fetches X/Twitter content through Apify and writes a single `data/feed-x.json` file. [prepare-digest.js](/Users/alexmini/Codex/oneAi/scripts/prepare-digest.js) loads that file and passes the normalized `x` array into the digest pipeline.

The important compatibility requirement is that `prepare-digest.js` should keep seeing the same top-level `feed-x.json` shape:

- `generatedAt`
- `lookbackHours`
- `x`
- `stats`

Each builder entry should also remain compatible with the current downstream contract:

- `source`
- `name`
- `handle`
- `bio`
- `avatarUrl`
- `tweets`

## Recommended Architecture

Use a three-layer X ingestion flow:

1. Chrome MCP acquisition layer
2. Structured archive layer
3. Compatibility export layer

This keeps the browser automation details isolated from the storage and downstream export formats.

### 1. Chrome MCP Acquisition Layer

Add an X-specific acquisition module that:

- opens the target account timeline in the logged-in Chrome session
- reads tweet cards for the daily window
- skips replies
- captures tweet text, timestamps, URLs, engagement stats, and image references
- returns normalized in-memory records for each account

The acquisition layer should not know about `feed-x.json` or about HTML templates. It should only return normalized account and tweet data plus fetch errors.

### 2. Structured Archive Layer

Persist each run into date-partitioned directories under `data/`:

- `data/x-raw/<YYYY-MM-DD>/<handle>.json`
- `data/x-html/<YYYY-MM-DD>/<handle>.html`
- `data/x-assets/<YYYY-MM-DD>/<handle>/...`

These outputs have distinct roles:

- `x-raw` is the reliable machine-readable archive of what the MCP fetch returned after normalization.
- `x-html` is a human-readable archive generated by the project from the structured data.
- `x-assets` stores the tweet images referenced by the structured data and HTML.

The system should generate HTML from the normalized data rather than storing any raw x.com page source. The HTML becomes a stable local artifact that can be inspected or reused later without re-running Chrome automation.

### 3. Compatibility Export Layer

Generate `data/feed-x.json` from the same normalized account data used to create the archives.

That file remains the interface consumed by [prepare-digest.js](/Users/alexmini/Codex/oneAi/scripts/prepare-digest.js), so the digest and video pipeline do not need to care whether the source was Apify or Chrome MCP.

## Data Model

### Normalized Account Record

Each account record should contain:

```json
{
  "source": "x",
  "name": "Andrej Karpathy",
  "handle": "karpathy",
  "bio": "",
  "avatarUrl": "",
  "tweets": []
}
```

The `name` and `handle` come from the configured builder list. `bio` and `avatarUrl` can be filled when available from the Chrome page content; otherwise they can be empty strings to preserve compatibility with the current downstream consumers.

### Normalized Tweet Record

Each tweet record should contain:

```json
{
  "id": "1900000000000000000",
  "text": "Tweet text",
  "createdAt": "2026-03-28T08:15:00.000Z",
  "url": "https://x.com/karpathy/status/1900000000000000000",
  "likes": 1234,
  "retweets": 120,
  "replies": 0,
  "images": [
    {
      "url": "https://pbs.twimg.com/media/example.jpg",
      "localPath": "x-assets/2026-03-28/karpathy/1900000000000000000-1.jpg",
      "alt": "optional alt text"
    }
  ]
}
```

Notes:

- `replies` remains present for compatibility even though reply tweets are excluded from the fetch set.
- `images` is a new field used by the archive layer and optional future downstream consumers.
- `localPath` should be stored relative to `data/`, not as an absolute path, so HTML generation and future relocation remain simple.

### Raw Archive Record

The raw archive file at `data/x-raw/<date>/<handle>.json` should contain:

- run metadata: `date`, `generatedAt`, `handle`, `name`
- account metadata: `bio`, `avatarUrl`
- `tweets`
- `errors`

This archive is project-owned normalized JSON, not a dump of an MCP payload. That keeps it stable even if the MCP response format changes.

## HTML Archive Format

Each account gets one HTML file for the day:

- `data/x-html/<YYYY-MM-DD>/<handle>.html`

The HTML should be generated from a local template and contain:

- account display name
- `@handle`
- archive date
- fetch timestamp
- count of archived tweets
- a list of tweet sections

Each tweet section should include:

- timestamp
- link to the original tweet URL
- tweet text
- zero or more locally referenced `<img>` tags for saved assets
- lightweight engagement metadata if present

The HTML should use relative paths to local assets so it remains portable within `data/`.

The HTML is intentionally simple and static. It is an archive artifact, not an application page.

## Daily Window Rules

The fetch is keyed by the local run date, for example `2026-03-28`.

For each account:

- collect tweets in the current daily window
- exclude replies
- keep original tweets
- do not include plain retweets by default

If the Chrome UI exposes quote-tweets as original authored posts with commentary, they may be kept, because they are effectively authored content rather than replies.

The daily window should be computed once at the start of the run and applied consistently to all accounts so the run is deterministic.

## File and Module Boundaries

The current [fetch-feeds.js](/Users/alexmini/Codex/oneAi/scripts/fetch-feeds.js) file is already doing too many jobs for the X path. The replacement should separate responsibilities into focused modules.

### Keep

- [fetch-feeds.js](/Users/alexmini/Codex/oneAi/scripts/fetch-feeds.js)
  - orchestrates the overall fetch
  - loads environment
  - ensures `data/` exists
  - fetches X, podcasts, and blogs
  - writes `feed-x.json`, `feed-podcasts.json`, and `feed-blogs.json`

### Add

- `scripts/lib/fetch-x-via-chrome-mcp.js`
  - opens and reads X timelines through Chrome MCP
  - normalizes account and tweet data
  - returns structured account results and errors

- `scripts/lib/x-storage.js`
  - builds date-based paths
  - writes `x-raw` JSON
  - writes images into `x-assets`
  - exposes relative-path helpers for archive output

- `scripts/lib/render-x-html.js`
  - converts normalized account data into project-owned HTML
  - references saved local image assets

This boundary keeps browser reading, persistence, and HTML rendering decoupled.

## Error Handling

The fetch should distinguish between run-level failures and account-level failures.

### Run-Level Failure

If the Chrome MCP bridge is not available or cannot start, the X fetch should fail loudly instead of silently falling back to Apify.

Reason: a silent fallback would hide a material source change and make the daily archives inconsistent with the user's request.

### Account-Level Failure

If one account cannot be read:

- log a failure for that account
- write an archive JSON file for that account with an empty `tweets` array and an `errors` entry
- continue with the remaining accounts

### Asset-Level Failure

If one image cannot be saved:

- keep the tweet
- omit that image from the HTML output or mark it unavailable in the raw JSON
- continue processing the remaining images and tweets

This keeps the archive usable even when media fetches are imperfect.

## Compatibility With Existing Digest Flow

[prepare-digest.js](/Users/alexmini/Codex/oneAi/scripts/prepare-digest.js) should continue loading `data/feed-x.json` without requiring structural changes in downstream consumers.

The only safe additive change is that tweet objects may now include an `images` field. Existing code that ignores unknown properties will continue working.

The export layer should therefore preserve:

- builder list semantics
- top-level feed stats
- existing tweet fields

## Verification Strategy

Because the repo has no automated test suite, verification should be command- and artifact-based.

### Fetch Verification

Run:

```bash
npm run fetch
```

Confirm that the run creates:

- `data/feed-x.json`
- `data/x-raw/<date>/...`
- `data/x-html/<date>/...`
- `data/x-assets/<date>/...`

Confirm at least one account archive contains:

- one or more tweets from the daily window
- no reply tweets
- local image references when images exist

### Downstream Verification

Run:

```bash
node scripts/generate-video.js --skip-fetch
```

Confirm the digest/video pipeline still reads the new `feed-x.json` successfully.

### Empty-Day Verification

If an account has no tweets in the daily window, it should still produce:

- a valid raw archive file
- a valid HTML archive file with zero tweets
- a valid builder entry in `feed-x.json`

This prevents downstream failures on quiet accounts.

## Risks and Non-Goals

### Risks

- Chrome UI structure on x.com can change, so the DOM extraction logic will need to be defensive.
- Media URLs may expire or require session context, so image saving should happen during the same fetch run.
- Browser-driven fetches will be slower than the prior API-based path.

### Non-Goals

- No change to podcasts or blogs.
- No redesign of the digest prompt flow.
- No attempt to archive full raw x.com pages.
- No headless cloud execution target in this phase.

## Implementation Readiness

This design is intentionally biased toward compatibility and inspectability:

- compatibility because `feed-x.json` remains the downstream contract
- inspectability because daily normalized JSON, HTML, and saved images all live under `data/`
- maintainability because browser acquisition is separated from storage and rendering

That makes the change large enough to justify a short implementation plan before touching the fetch pipeline.
